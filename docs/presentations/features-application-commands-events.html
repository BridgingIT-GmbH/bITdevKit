Application Commands, Queries, and Events

20-slide field guide for using IRequester and INotifier cleanly in app-layer code.

Based on features-application-commands-queries.md, features-application-events.md, and features-requester-notifier.md.

The Three Shapes

01 / COMMAND

One owner. State change.

Use when the app is asked to do work and give back one explicit outcome.

  • Write intent
  • Exactly one handler
  • Result<Unit> or Result<T>

02 / QUERY

One owner. Typed read.

Use when the app is asked to answer a question without changing state.

  • Read intent
  • Exactly one handler
  • Usually Result<T>

03 / EVENT

One publisher. Many reactions.

Use when the app announces something and subscribers decide what happens next.

  • Fan-out intent
  • One or many handlers
  • Aggregated Result

Part 1

Commands

State-changing requests with one handler and one outcome.

Command Mental Model

WHEN TO USE

Commands are application writes

  • The caller wants business work executed now.
  • The operation has one clear owner.
  • You want an explicit success/failure contract.
  • The next step depends on the returned result.

SMELL

A command is not a broadcast

If the main behavior is "tell other parts of the app something happened", that is an event. A command should still feel like an API operation with one primary handler.

Authoring A Command

[Command] / source-generated request

[Command]
public partial class CustomerRenameCommand
{
    [ValidateNotEmptyGuid("CustomerId is required.")]
    public string CustomerId { get; init; }

    [ValidateNotEmpty("DisplayName is required.")]
    public string DisplayName { get; init; }

    [Handle]
    private async Task<Result<Unit>> HandleAsync(
        ICustomerRepository repository,
        CancellationToken cancellationToken)
    {
        await repository.RenameAsync(CustomerId, DisplayName, cancellationToken);
        return Success();
    }
}

GENERATOR GIVES YOU

  • Handler plumbing
  • Success(...) and Failure(...)
  • Validation bridge
  • Request base type when needed

RULES

  • One instance [Handle] method
  • DI parameters are allowed
  • SendOptions and CancellationToken are supported

GOOD FIT

  • Create, update, delete, schedule, assign
  • Use cases where writes are the point

RETURN STYLE

  • Result<Unit> when the caller only needs completion
  • Result<T> when the caller needs a small response model

Part 2

Queries

Typed reads with one handler and no state change.

Query Mental Model

WHEN TO USE

Queries answer questions

  • The caller wants data, not a write.
  • You still want a single owner.
  • The result shape matters to the caller.
  • Pipeline behaviors still apply around the handler.

KEY DISTINCTION

A query shares infrastructure, not intent

Queries and commands both use IRequester, but they should not be treated as interchangeable in design. A query is a read contract first.

Authoring A Query

[Query] / source-generated request

[Query]
public partial class CustomerDetailsQuery
{
    [ValidateNotEmptyGuid("CustomerId is required.")]
    public string CustomerId { get; init; }

    [Handle]
    private async Task<Result<CustomerDetailsModel>> HandleAsync(
        ICustomerReadRepository repository,
        CancellationToken cancellationToken)
    {
        return await repository.GetDetailsAsync(CustomerId, cancellationToken);
    }
}

QUERY RULES

  • One instance [Handle] method
  • Return Result<T> for the response model
  • Property validation still works the same

PRACTICE

  • Prefer query-specific response models
  • Keep side effects out of the read path
  • Let the handler own query composition

Query Flow

1

Caller asks a question

SendAsync(query)

->
2

Requester pipeline

validation and policies still apply

->
3

Single query handler

builds typed read result

->
4

Result<T>

data for the caller

GOOD FIT

  • Details screens
  • Search results
  • Availability checks

NOT THE PLACE FOR

  • Writes, retries with side effects, or notifications
  • Hidden cache invalidation or audit mutations

Part 3

Application Events

App-layer pub/sub through INotifier.

Event Mental Model

WHEN TO USE

Events announce, they do not ask

  • The publisher should not know all subscribers.
  • Multiple follow-up reactions can grow independently.
  • The main value is fan-out inside the application layer.

MIXED MODE

Generated and manual handlers can coexist

Use generated [Handle] methods for the default path, then add manual INotificationHandler<TEvent> classes when a subscriber deserves its own boundary.

N

handlers

FAN OUT

intent

Result

Authoring An Event

[Event] / multiple generated handlers

[Event]
[HandlerRetry(2, 100)]
public partial class UserRegisteredEvent
{
    [ValidateNotEmptyGuid]
    public string UserId { get; init; }

    [ValidateEmail]
    public string Email { get; init; }

    [Handle]
    private Result Audit() => Success();

    [Handle]
    private async Task<Result> SendWelcomeEmailAsync(
        INotificationService<EmailMessage> notifications,
        CancellationToken cancellationToken)
    {
        return Success();
    }
}

EVENT RULES

  • One or more [Handle] methods
  • Validation runs once per publish
  • Class-level policy attributes apply to generated handlers

RESULT

  • Each generated handle becomes its own notifier handler
  • Publish returns aggregated Result

Event Fan-Out

1

Publisher calls PublishAsync(...)

endpoint, app service, or request handler

2

Validation runs once

property rules and optional [Validate]

3a

Generated handlers run

one generated subscriber per [Handle]

3b

Manual handlers run too

extra INotificationHandler<TEvent> classes still participate

4

Notifier returns aggregated Result

publisher gets one outcome, subscribers stay decoupled

Generated Vs Manual Event Handlers

PREFER GENERATED

  • The behavior belongs next to the event authoring surface.
  • You want the shortest path from message to reaction.
  • The event and its default reactions evolve together.

PREFER MANUAL

  • The subscriber deserves its own file and name.
  • The reaction belongs to another module boundary.
  • The handler has enough weight to stand alone.

Generated event handlers reduce boilerplate. Manual handlers preserve explicit boundaries. Use both when the design asks for both.

Part 4

Developer Playbook

How to pick the right shape and keep the boundaries sharp.

Commands Vs Queries Vs Events

Need to change state

Command

Need typed data back

Query

Need many in-process reactions

Application Event

Business fact raised by an aggregate

Domain Event

Cross-process durable delivery

Messaging / Outbox

Return Types Matter

COMMAND

Result<Unit> or Result<T>

Return completion or a small post-write response.

QUERY

Result<T>

Return the read model the caller actually needs.

EVENT

Result

Notifier aggregates subscriber outcomes for the publisher.

The return type should tell the truth about the interaction shape. That is part of the API design, not just plumbing.

Wiring Checklist

SETUP

  • Register AddRequester() for commands and queries.
  • Register AddNotifier() for application events.
  • Call AddHandlers() so generated and manual handlers are discovered.

PIPELINE

  • Add validation centrally.
  • Add retry, timeout, auth, and related behaviors centrally.
  • Keep handlers focused on business logic, not cross-cutting plumbing.
