Application Commands And Queries

20-slide field guide for using IRequester cleanly for application-layer writes and reads.

Based on features-application-commands-queries.md, with related references to features-requester-notifier.md, features-results.md, and common-mapping.md.

The Two Shapes

01 / COMMAND

One owner. State change.

Use when the app is asked to do work and return one explicit write outcome.

  • Write intent
  • Exactly one handler
  • Result<Unit> or Result<T>

02 / QUERY

One owner. Typed read.

Use when the app is asked a question and should return data without changing state.

  • Read intent
  • Exactly one handler
  • Usually Result<T>

03 / SHARED REQUESTER

Infrastructure is shared, intent is not

Commands and queries both use IRequester, but they should still tell the truth about whether the caller wants a write or a read.

  • Validation and retry
  • Type-safe dispatch
  • Focused handler boundaries

Part 1

Commands

State-changing requests with one handler and one outcome.

Command Mental Model

WHEN TO USE

Commands are application writes

  • The caller wants business work executed now.
  • The operation has one clear owner.
  • You want an explicit success/failure contract.
  • The next step depends on the returned result.

SMELL

A command is not a read shortcut

If the main need is to return data without changing state, that is a query. A command should still feel like an owned business operation.

Authoring A Command

[Command] / source-generated request

[Command]
public partial class CustomerRenameCommand
{
    [ValidateNotEmptyGuid("CustomerId is required.")]
    public string CustomerId { get; init; }

    [ValidateNotEmpty("DisplayName is required.")]
    public string DisplayName { get; init; }

    [Handle]
    private async Task<Result<Unit>> HandleAsync(
        ICustomerRepository repository,
        CancellationToken cancellationToken)
    {
        await repository.RenameAsync(CustomerId, DisplayName, cancellationToken);
        return Success();
    }
}

GENERATOR GIVES YOU

  • Handler plumbing
  • Success(...) and Failure(...)
  • Validation bridge
  • Request base type when needed

RULES

  • One instance [Handle] method
  • DI parameters are allowed
  • SendOptions and CancellationToken are supported

RETURN STYLE

  • Result<Unit> when the caller only needs completion
  • Result<T> when the caller needs a small post-write response model

Part 2

Queries

Typed reads with one handler and no state change.

Query Mental Model

WHEN TO USE

Queries answer questions

  • The caller wants data, not a write.
  • You still want a single owner.
  • The result shape matters to the caller.
  • Pipeline behaviors still apply around the handler.

KEY DISTINCTION

A query shares infrastructure, not intent

Queries and commands both use IRequester, but they should not be treated as interchangeable. A query is a read contract first.

Authoring A Query

[Query] / source-generated request

[Query]
public partial class CustomerDetailsQuery
{
    [ValidateNotEmptyGuid("CustomerId is required.")]
    public string CustomerId { get; init; }

    [Handle]
    private async Task<Result<CustomerDetailsModel>> HandleAsync(
        IMapper mapper,
        ICustomerReadRepository repository,
        CancellationToken cancellationToken)
    {
        return await repository.GetDetailsAsync(CustomerId, cancellationToken);
    }
}

QUERY RULES

  • One instance [Handle] method
  • Return Result<T> for the response model
  • DI parameters are allowed just like commands

PRACTICE

  • Prefer query-specific response models
  • Keep side effects out of the read path
  • Let the handler own query composition

Query Flow

data for the caller

GOOD FIT

  • Details screens
  • Search results
  • Availability checks

NOT THE PLACE FOR

  • Writes or hidden mutations
  • Subscriber-style fan-out work
  • Side effects disguised as reads

Part 3

Shared Requester Model

Validation, DI, policies, and result design for both message shapes.

Shared Requester Pipeline

VALIDATION

Fail early on bad input

property attributes or explicit [Validate] methods

RETRY

Recover transient faults

policy behavior, not handler boilerplate

TIMEOUT

Bound execution

handler policies and builder defaults

AUTH

Protect the request surface

authorization and related cross-cutting concerns

The main value of IRequester is not just dispatch. It is the consistent execution model around handlers.

Handle Methods, DI, And Policies

HANDLE METHOD INPUTS

  • Request properties are accessed through this
  • DI services can be declared as parameters and are resolved automatically
  • CancellationToken and SendOptions are supported

POLICIES

  • Retry, timeout, authorization, transaction, and related policies can be attached declaratively
  • Keep technical plumbing out of the handler body
  • The goal is smaller, focused handlers

DI inside a generated request handler method

[Handle]
private async Task<Result<CustomerDetailsModel>> HandleAsync(
    IMapper mapper,
    ICustomerReadRepository repository,
    SendOptions options,
    CancellationToken cancellationToken)
{
    var customer = await repository.GetDetailsAsync(CustomerId, cancellationToken);
    return customer;
}

Validation Model

SIMPLE RULES

[Command]
public partial class CustomerRenameCommand
{
    [ValidateNotEmptyGuid("CustomerId is required.")]
    public string CustomerId { get; init; }

    [ValidateNotEmpty("Display name is required.")]
    [ValidateLength(3, 100)]
    public string DisplayName { get; init; }

    [Handle]
    private Result<Unit> Handle() => Success();
}

RICHER RULES

[Validate]
private static void Validate(
    InlineValidator<CustomerImportCommand> validator)
{
    validator.RuleFor(x => x.Emails)
        .Must(x => x.Count <= 100)
        .WithMessage("A maximum of 100 email addresses is allowed.");
}

Return Types Matter

COMMAND

Result<Unit> or Result<T>

Return completion or a small post-write response model. The type should tell the truth about the write contract.

QUERY

Result<T>

Return the read model the caller actually needs. The type should reflect the data contract, not internal domain shape by accident.

Part 4

Developer Playbook

How to choose the right shape and keep the boundaries sharp.

Commands Vs Queries

Need to change state

Command

Need typed data back

Query

Need one clear owner

Both

Need fan-out to many subscribers

Application Event, not this deck

Wiring Checklist

SETUP

  • Register AddRequester()
  • Call AddHandlers() so generated and manual handlers are discovered
  • Add the code generation package where source-generated requests live

PIPELINE

  • Add validation centrally
  • Add retry, timeout, and related behaviors centrally
  • Keep handlers focused on business logic, not cross-cutting plumbing

Common Design Rules

COMMAND

  • Do not hide reads as writes
  • Keep one clear owner
  • Return the smallest truthful outcome

QUERY

  • Do not hide writes in the read path
  • Prefer response models over leaking persistence shape
  • Keep side effects out of query handlers

BOUNDARY

  • Use application events for fan-out, not request/response
  • Use domain events for aggregate-originated business facts
