Application DataPorter

Guide to importing, exporting, validating, and streaming structured application data through format-agnostic DataPorter APIs.

Based on features-application-dataporter.md, with related references to features-results.md and common-serialization.md.

DataPorter At A Glance

01 / API SURFACE

Import and export through application services

  • IDataExporter and IDataImporter
  • Typed results and errors
  • Fluent per-call options

02 / CONFIGURATION

Attributes, profiles, and runtime options merge

  • Simple DTO annotations
  • Profiles for richer mapping and styling
  • Provider defaults plus per-call overrides

03 / FORMAT FLEXIBILITY

One orchestration model, many providers

  • Excel, CSV, typed CSV, JSON, XML, PDF
  • Streaming where supported
  • Custom providers can plug in

Runtime Shape

1

Application code calls importer or exporter

IDataImporter / IDataExporter

->
2

DataPorter builds effective configuration

profiles, attributes, runtime options

->
3

Provider executes the format work

CSV, Excel, JSON, XML, PDF, or custom

->
4

Typed result returns

export/import result, rows, errors, diagnostics

KEY IDEA

Application code stays on typed import/export APIs while DataPorter handles provider selection, configuration precedence, validation, progress reporting, and compression around the edges.

Part 1

Getting Started

Register providers, choose a format, and call import or export through the shared APIs.

Service Registration

Provider registration through the builder

services.AddDataPorter(configuration)
    .WithExcel(c => c.UseTableFormatting = true)
    .WithCsv(c => c.Delimiter = ",")
    .WithCsvTyped(c => c.Delimiter = ",")
    .WithJson()
    .WithXml()
    .WithPdf();

services.AddDataPorter(configuration)
    .WithCsv()
    .AddProvider<EdiX12DataPorterProvider>();

DEFAULT MOVE

  • Register the built-in formats your app actually needs
  • Use builder config for application-wide defaults
  • Add custom providers only for genuinely new external formats

BOUNDARY

Keep DataPorter registration in composition root code. Application services should depend on IDataExporter or IDataImporter, not on provider implementations.

Basic Export Flow

Export with fluent options

public class OrderExportService
{
    private readonly IDataExporter exporter;

    public OrderExportService(IDataExporter exporter)
    {
        this.exporter = exporter;
    }

    public async Task ExportOrdersAsync(IEnumerable<Order> orders, Stream output)
    {
        var result = await this.exporter.ExportAsync(
            orders,
            output,
            o => o.AsExcel()
                .WithProgress(report =>
                {
                    Console.WriteLine($"Exported {report.ProcessedRows} rows");
                }));

        if (result.IsSuccess)
        {
            Console.WriteLine($"Exported {result.Value.TotalRows} rows");
        }
    }
}

FLOW

  • Pass source data and target stream
  • Select a format with the fluent options builder
  • Read a typed export result instead of guessing outcome from exceptions

GOOD FIT

Use export when the app needs to produce partner feeds, user downloads, reports, templates, or structured outbound payloads without hand-writing format logic for each case.

Basic Import Flow

Import with culture and progress options

public async Task<IEnumerable<Order>> ImportOrdersAsync(Stream input)
{
    var result = await this.importer.ImportAsync<Order>(
        input,
        o => o.AsCsv()
            .WithCulture(new CultureInfo("de-DE"))
            .WithProgress(report =>
            {
                Console.WriteLine($"Processed {report.ProcessedRows} rows");
            }));

    if (result.IsSuccess && !result.Value.HasErrors)
    {
        return result.Value.Data;
    }

    if (result.IsSuccess)
    {
        foreach (var error in result.Value.Errors)
        {
            Console.WriteLine($"Row {error.RowNumber}: {error.Message}");
        }
    }

    return [];
}

FLOW

  • Pass the input stream and target type
  • Set culture and other per-run parsing options
  • Handle both operation failure and row-level validation feedback

IMPORT SHAPE

Successful import can still carry row errors depending on validation behavior. Treat the returned import result as the full picture, not just a pass/fail flag.

Runtime Options That Matter

PROGRESS

Observe long-running operations

export and import can report row counts, bytes, percent, and validation progress

COMPRESSION

Wrap payloads as .gz or single-entry ZIP

use DataPorter compression when the artifact itself should be compressed

TEMPLATES

Generate import-ready guidance artifacts

Excel, CSV, typed CSV, JSON, and XML template support

MULTI-DATASET

Export several data sets in one call

works as sheets, sections, or document groups depending on provider

Provider config sets application-wide defaults. Per-call options are where one specific import or export request becomes tailored to a concrete workflow.

Part 2

Configuration

Choose attributes for simple DTO cases or profiles for richer mapping, validation, and formatting rules.

Attribute-Based Configuration

Attributes directly on the DTO

[DataPorterSheet("Products")]
public class ProductDto
{
    [DataPorterColumn("Product ID", Order = 0)]
    public string Id { get; set; }

    [DataPorterColumn("Name", Order = 1, Required = true)]
    public string Name { get; set; }

    [DataPorterColumn("Price", Format = "C2")]
    public decimal Price { get; set; }

    [DataPorterIgnore]
    public string InternalCode { get; set; }
}

USE ATTRIBUTES WHEN

  • The mapping is simple and local to one DTO
  • You only need straightforward column naming and ordering
  • You want the metadata close to the type definition

AVAILABLE SHAPES

  • [DataPorterSheet]
  • [DataPorterColumn]
  • [DataPorterIgnore]
  • [DataPorterConverter] and [DataPorterValidation]

Profile-Based Configuration

Profiles for richer export and import rules

public class OrderExportProfile : ExportProfileBase<Order>
{
    protected override void Configure()
    {
        ToSheet("Orders");

        ForColumn(o => o.TotalAmount)
            .HasName("Total")
            .HasFormat("C2")
            .StyleWhen(amount => amount > 1000, style => style.Bold());
    }
}

public class OrderImportProfile : ImportProfileBase<Order>
{
    protected override void Configure()
    {
        FromSheet("Orders");
        OnValidationFailure(ImportValidationBehavior.CollectErrors);

        ForColumn(o => o.CustomerName)
            .FromHeader("Customer")
            .IsRequired();
    }
}

USE PROFILES WHEN

  • You need full control over formatting, parsing, validation, and styling
  • Several import/export cases share the same mapping rules
  • You want configuration outside the DTO or entity type

REGISTRATION

Register explicit profiles or scan assemblies through the DataPorter builder. Profiles and attributes are merged with runtime options before the provider sees the final configuration.

Choosing A Format Provider

EXCEL / PDF

  • Best for human-facing downloads and reports
  • Excel adds styling, widths, frozen headers, conditional formatting
  • PDF is export-only and presentation-oriented

CSV / TYPED CSV

  • Best for tabular exchange and partner feeds
  • CSV can flatten nested members and one nested collection
  • Typed CSV models object graphs with stable row schemas

JSON / XML

  • Best for document-style payloads and integration boundaries
  • Incremental read/write support for large streams
  • Custom providers are available when none of the built-ins fit

RULE OF THUMB

Pick the provider for the external representation, not for a mapping preference. Mapping and validation belong in profiles, attributes, and converters.

Part 3

Advanced Usage

Stream large payloads, customize value conversion, and decide how import validation should behave.

Streaming, Templates, And Validation Without Import

STREAMING

  • ExportAsync(IAsyncEnumerable<T>...) writes items incrementally
  • ImportAsyncEnumerable<T>(...) yields items or row results incrementally
  • Use it for large files and HTTP streaming scenarios

OTHER HIGH-VALUE CALLS

  • GenerateTemplateAsync<T>(...) for guided import artifacts
  • ValidateAsync<T>(...) for preflight checks without importing
  • Multi-dataset export for sheets or grouped sections

For grouped CSV and typed CSV streaming imports, logical rows for the same aggregate must stay contiguous. If they reappear later, the stream cannot be reconstructed safely.

Converters And Validation Behaviors

Custom converter and validation behavior

ForColumn(o => o.Status)
    .UseConverter(new StringMapConverter<OrderStatus>
    {
        ImportMappings = new Dictionary<string, OrderStatus>
        {
            ["P"] = OrderStatus.Pending,
            ["S"] = OrderStatus.Shipped
        },
        ExportMappings = new Dictionary<OrderStatus, string>
        {
            [OrderStatus.Pending] = "P",
            [OrderStatus.Shipped] = "S"
        }
    });

OnValidationFailure(ImportValidationBehavior.CollectErrors);

CONVERTERS

  • Translate one property value between .NET shape and external representation
  • Use them for external codes, localized formats, trimming, enum display names, or smart-enum mapping
  • Keep provider code focused on transport, not business conversion rules

IMPORT VALIDATION

  • CollectErrors keeps valid rows and reports invalid ones
  • SkipRow ignores invalid rows and continues
  • StopImport fails on the first validation problem
