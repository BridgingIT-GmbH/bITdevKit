Application Events

20-slide field guide for publishing application-layer events through INotifier with generated handlers, explicit results, and clear boundaries.

Based on features-application-events.md, features-requester-notifier.md, and features-domain-events.md.

Event Shape

01 / PUBLISH

One publisher. Many in-process reactions.

The application layer announces something and subscribers decide what happens next.

  • Publish through INotifier
  • One or many handlers
  • Publisher stays decoupled

02 / PIPELINE

Validation and resiliency stay shared

Application events reuse the same notifier pipeline infrastructure as the rest of the feature set.

  • Validation, retry, timeout, authorization
  • Generated and manual handlers together
  • Class-level policy attributes

03 / RESULT

Publication still returns an explicit outcome

Handlers do not disappear into hidden background behavior by default.

  • Handlers return Result
  • Notifier aggregates the outcome
  • Failures stay visible at the call site

Part 1

Event Model

What application events are, when they fit, and how they are authored.

Event Mental Model

WHEN TO USE

Application events announce follow-up work

  • A command or endpoint should trigger several in-process reactions.
  • The publisher should not know concrete subscribers.
  • The work belongs to the application layer, not the aggregate.
  • Handlers should share the same policy pipeline as other app messages.

NOT THE SAME AS

Do not collapse all signals into one model

Application events are not domain events and they are not durable external messaging. They are app-layer pub/sub inside one process.

Authoring An Event

[Event] / source-generated notification

[Event]
public partial class UserRegisteredEvent
{
    [ValidateNotEmptyGuid("UserId is required.")]
    public string UserId { get; init; }

    [ValidateNotEmpty("Email is required.")]
    [ValidateEmail("Email must be valid.")]
    public string Email { get; init; }

    [Handle]
    private Result Audit()
    {
        Console.WriteLine($"Audit registration for {UserId}");
        return Success();
    }

    [Handle]
    private async Task<Result> SendWelcomeEmailAsync(
        INotificationService<EmailMessage> notifications,
        CancellationToken cancellationToken)
    {
        var sendResult = await notifications.SendAsync(
            new EmailMessage { To = [Email] },
            cancellationToken: cancellationToken);

        return sendResult.IsSuccess ? Success() : Failure("Welcome email could not be sent.");
    }
}

GENERATOR GIVES YOU

  • One generated handler per [Handle] method
  • NotificationBase if you do not inherit it manually
  • Success(...) and Failure(...)
  • Validation bridge for property rules and [Validate]

PAYOFF

  • One partial type can fan out to many subscribers
  • Default reactions stay close to the event surface
  • Boilerplate drops sharply without changing runtime behavior

Handle Methods And DI

HANDLE METHOD RULES

  • One or more instance [Handle] methods are allowed
  • Methods must return Result or Task<Result>
  • Event data is accessed directly through this

DI PARAMETERS

  • DI services can be declared as parameters and are resolved automatically
  • CancellationToken and PublishOptions are supported too
  • This follows the same ergonomic pattern as commands and queries

DI inside a generated event handler method

[Handle]
private async Task<Result> InvalidateCacheAsync(
    ICache cache,
    PublishOptions options,
    CancellationToken cancellationToken)
{
    await cache.RemoveAsync($"user:{UserId}", cancellationToken);
    return Success();
}

KEY IDEA

Application events keep the publisher thin while preserving the same policy pipeline and explicit outcome model used elsewhere in the application layer.

Generated And Manual Handlers Together

Manual handler beside generated ones

public class UserRegisteredMetricsHandler
    : NotificationHandlerBase<UserRegisteredEvent>
{
    protected override Task<Result> HandleAsync(
        UserRegisteredEvent notification,
        PublishOptions options,
        CancellationToken cancellationToken)
    {
        Console.WriteLine($"Metrics tracked for {notification.UserId}");
        return Task.FromResult(Result.Success());
    }
}

PREFER GENERATED

  • The default reactions belong next to the event surface
  • You want the lowest ceremony authoring model
  • The event and its default subscribers evolve together

PREFER MANUAL

  • The subscriber deserves its own file and name
  • The reaction belongs to a different module boundary
  • The handler has enough weight to stand alone

Publishing From Application Code

Publishing through INotifier

public class RegistrationService(INotifier notifier)
{
    public async Task<Result> RegisterAsync(
        string userId,
        string email,
        CancellationToken cancellationToken)
    {
        // application logic omitted

        return (Result)await notifier.PublishAsync(
            new UserRegisteredEvent
            {
                UserId = userId,
                Email = email
            },
            cancellationToken: cancellationToken);
    }
}

CALL-SITE TRUTH

  • The publisher only depends on INotifier
  • Subscriber details stay hidden from the publisher
  • The returned Result keeps follow-up failure explicit

DO NOT ASSUME

  • Publishing is not the same as durable background delivery
  • It is still an application-layer interaction with a real outcome

Part 3

Validation And Boundaries

Validation shapes, handler policies, and the boundary line versus domain events and messaging.

Validation Model

SIMPLE RULES

[Event]
public partial class InvoiceApprovedEvent
{
    [ValidateNotEmptyGuid("InvoiceId is required.")]
    public string InvoiceId { get; init; }

    [ValidateNotEmpty("ApprovedBy is required.")]
    public string ApprovedBy { get; init; }

    [Handle]
    private Result Handle() => Success();
}

RICHER RULES

[Validate]
private static void Validate(
    InlineValidator<CustomerImportCompletedEvent> validator)
{
    validator.RuleFor(x => x.ImportedCustomerIds)
        .Must(ids => ids.Count <= 1000)
        .WithMessage("A maximum of 1000 imported customers is allowed.");
}

The generated validator plugs into the normal notifier validation behavior, so event authoring stays aligned with the rest of the application message model.

Application Events Vs Domain Events

Aggregate records a business-significant state transition

Domain event

Application layer decides to trigger follow-up reactions

Application event

Publisher should not know which in-process subscribers react

Application event

Signal originates as part of the aggregate model itself

Domain event

RULE OF THUMB

CustomerCreatedDomainEvent is aggregate-originated. WelcomeEmailRequestedEvent is application-layer orchestration after the main use case completes.

Application Events Vs Messaging

APPLICATION EVENTS

  • In-process reactions
  • Coupled to the running application instance
  • Fast orchestration and side effects
  • Explicit notifier result back to the publisher

MESSAGING

  • Durable cross-process delivery
  • Transport concerns and reliability semantics
  • Use when the signal must survive process boundaries
  • Often pairs with outbox or broker infrastructure

Part 4

Developer Playbook

Rules, setup, and practical decision guidance.

Authoring Rules

TYPE SHAPE

  • [Event] requires a top-level, non-generic, partial class
  • If the event does not inherit NotificationBase, the generator adds it

HANDLE METHODS

  • One or more [Handle] methods are allowed
  • Return Result or Task<Result>
  • DI services, PublishOptions, and CancellationToken are supported

COMPATIBILITY

  • Manual INotificationHandler<TEvent> subscribers remain fully supported
  • Generated and manual handlers still share the same runtime model

Wiring Checklist

SETUP

  • Register AddNotifier()
  • Call AddHandlers() so generated and manual handlers are discovered
  • Add the code generation package where source-generated events live

PIPELINE

  • Add validation centrally
  • Add retry, timeout, and related policies centrally
  • Keep handlers focused on business reactions, not plumbing

When Not To Use Application Events

PICK SOMETHING ELSE WHEN

  • The operation has one clear owner and should behave like an API call
  • The signal originates from aggregate state change inside the domain model
  • Delivery must be durable across process boundaries

PRACTICE

  • Use commands for owned business operations
  • Use domain events for aggregate-originated business facts
  • Use messaging for external or durable delivery needs

Application events are strongest when the primary need is app-layer fan-out with explicit results, not just “some sort of event.”
