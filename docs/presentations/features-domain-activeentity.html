Domain ActiveEntity

20-slide field guide for using entity-centric CRUD, providers, behaviors, and scoped operations without falling into accidental complexity.

Based on features-domain-activeentity.md, plus features-results.md and features-domain-specifications.md.

What ActiveEntity Is

01 / ENTITY-CENTRIC

Persistence methods live on the entity

  • InsertAsync()
  • UpdateAsync()
  • DeleteAsync()
  • Static find/query helpers

02 / PROVIDER-BASED

Persistence is still pluggable

  • EF Core provider
  • InMemory provider
  • Custom data source support

03 / RESULT-DRIVEN

Operations return explicit outcomes

  • Result
  • Result<T>
  • ResultPaged<T>

Where It Fits

GOOD FIT

Choose ActiveEntity when simplicity is the point

  • CRUD-heavy modules
  • Rapid prototypes and admin workflows
  • Domain-rich apps that still want direct entity-centric operations
  • Tests that benefit from provider swapping

NOT A UNIVERSAL DEFAULT

Do not force it everywhere

  • Complex reporting or batch workloads may fit repositories better
  • Cross-entity orchestration still belongs in app services or commands
  • High-complexity persistence models still need architectural judgment

Part 1

Basic Model

Entity definition, setup, and the core CRUD path.

Defining An ActiveEntity

Entity definition

[TypedEntityId<Guid>]
public class Customer : ActiveEntity<Customer, CustomerId>,
    IAuditable,
    IConcurrency
{
    public string FirstName { get; set; }
    public string LastName { get; set; }
    public string Title { get; set; }
    public EmailAddressStub Email { get; set; }
    public int Visits { get; set; } = 0;
}

BASE TYPE

  • Inherits from ActiveEntity<TEntity, TId>
  • Still builds on the standard entity/id model
  • Supports typed IDs cleanly

OPTIONAL INTERFACES

  • IAuditable for audit state
  • IConcurrency for concurrency support

Wiring Providers And Behaviors

DI setup

services.AddDbContext<DbContext>(options =>
    options.UseSqlServer(connectionString));

services.AddActiveEntity(cfg =>
{
    cfg.For<Customer, CustomerId>()
        .UseEntityFrameworkProvider(o => o.UseContext<DbContext>())
        .AddLoggingBehavior()
        .AddAuditStateBehavior(
            new ActiveEntityEntityAuditStateBehaviorOptions
            {
                SoftDeleteEnabled = true
            });
});

app.UseActiveEntity(app.Services);

PER ENTITY

  • Choose provider per entity
  • Add behaviors in registration order
  • Swap data stores without rewriting entity code

MENTAL MODEL

The entity surface feels direct, but the actual persistence work is delegated through provider and behavior infrastructure resolved from DI.

Core CRUD Surface

INSERT

New entity

customer.InsertAsync()

UPDATE

Loaded entity

customer.UpdateAsync()

DELETE

Instance or id

customer.DeleteAsync() or static delete

READ

Static helpers

FindOneAsync, FindAllAsync, paging, count, exists

PATTERN

The entity carries the public API, while providers and behaviors carry the infrastructure details.

KEY IDEA

The provider does persistence. Behaviors add cross-cutting concerns. The entity remains the developer-facing entry point.

Behaviors Are The Extension Layer

LOGGING

Trace operations

observe CRUD outcomes and state transitions

AUDIT

Track changes

audit state and optional soft delete

DOMAIN EVENTS

Publish around persistence

before or after successful operations

VALIDATION

Stop bad writes

DataAnnotations or FluentValidation

Use behaviors for reusable cross-cutting concerns. Do not bury those concerns inside every entity method manually.

Validation Options

DATA ANNOTATIONS

  • [Required]
  • [MinLength], [MaxLength]
  • [Range], [EmailAddress]
  • Good for simple entity property rules

FLUENTVALIDATION

  • Attach validators per operation type
  • Insert/update/delete can each have different rules
  • Good for richer business validation

PRACTICE

Keep validation where it best fits: simple property rules in annotations, richer or operation-specific rules in dedicated validators.

WithContextAsync And WithTransactionAsync

WITH CONTEXT

Reuse one provider scope

  • Use when multiple operations must share one consistent context
  • Does not automatically imply a transaction
  • Pass the provided ctx to nested operations

WITH TRANSACTION

Run atomically

  • Uses one transaction-aware context
  • Commit on success, rollback on failure
  • All operations inside the block must use the provided ctx

This scope discipline is one of the most important details in the feature. It prevents mismatched providers, disposed scopes, and inconsistent DbContext usage.

Part 3

Query Features

Convention finders, specifications, DSL, and forwarders.

Opt-In Features Per Entity

[ActiveEntityFeatures]

[TypedEntityId<Guid>]
[ActiveEntityFeatures(
    ActiveEntityFeatures.ConventionFinders |
    ActiveEntityFeatures.QueryDsl)]
public partial class Customer
    : ActiveEntity<Customer, CustomerId>
{
}

AVAILABLE FLAGS

  • Forwarders
  • ConventionFinders
  • Specifications
  • QueryDsl

DEFAULT

If no parameters are provided, all available features are enabled by default.

Convention Finders And Specifications

CONVENTION FINDERS

  • FindAllByLastNameAsync("Doe")
  • FindOneByEmailAsync(...)
  • Best for common lookups with clear property names

SPECIFICATIONS

  • Customer.Specifications.LastNameEquals("Doe")
  • Composable with And and Or
  • Best for reusable query criteria

For the underlying specification model beyond the generated helpers, see features-domain-specifications.md.

Query DSL And Custom Forwarders

QUERY DSL

  • Start with Customer.Query()
  • Chain Where, Include, OrderBy, Skip, Take
  • Finish with ToListAsync(), ToPagedListAsync(), CountAsync(), or projections

FORWARDERS

  • Expose custom static queries as if they were built into the entity
  • Useful for reusable domain-specific query shortcuts
  • Keep the caller syntax direct without hiding the implementation pattern

Part 4

Boundaries And Tradeoffs

Callbacks, cautions, repository comparison, and adoption guidance.

Lifecycle Callbacks

Entity-intrinsic rule

protected override async Task<Result>
OnBeforeInsertAsync(
    ActiveEntityContext<Order, OrderId> context,
    CancellationToken ct)
{
    var customerResult = await Customer.FindOneAsync(
        context,
        e => e.Id == customerId && e.IsActive,
        null,
        ct);

    if (customerResult.IsFailure)
    {
        return Result.Failure(
            "Order cannot be placed for an inactive or non-existent customer.");
    }

    return Result.Success();
}

GOOD USE

  • Simple rules intrinsic to the entity
  • Hooks directly into insert/update/delete lifecycle
  • Can stop an operation with Result.Failure

CAUTION

  • Do not turn callbacks into a dumping ground
  • Cross-cutting concerns belong in behaviors
  • Broader workflows belong in app services or commands

ActiveEntity Vs Repository

Fast, direct CRUD surface on the entity

ActiveEntity

Clear persistence abstraction boundary

Repository

Per-entity provider flexibility with minimal ceremony

ActiveEntity

Very complex queries or heavy data access orchestration

Repository

Hybrid use in one solution

Valid

The docs explicitly frame this as a tradeoff, not a winner-takes-all pattern decision.

ADOPTION SHORTLIST

  • Use it where direct entity-centric CRUD improves clarity
  • Configure providers and behaviors intentionally
  • Reuse ctx correctly in scoped composite operations
  • Do not treat it as a universal repository replacement

ActiveEntity is strongest when the directness is honest: simple CRUD, explicit results, pluggable persistence, and scope-safe execution.
