Pipelines

Guide to defining ordered multi-step workflows with shared context, explicit control flow, hooks, behaviors, and tracked execution.

Based on features-pipelines.md, with related references to features-results.md and features-rules.md.

Pipelines At A Glance

01 / WORKFLOW

Explicit ordered steps

  • Known step sequence
  • Shared context object
  • One carried result through the flow

02 / CONTROL

Each step decides what happens next

  • Continue
  • Retry, Break, Terminate
  • Consistent failure handling

03 / OBSERVABILITY

Built for tracing and background tracking

  • Hooks and behaviors
  • Progress and logging
  • Background execution snapshots

Where Pipelines Fit

GOOD FIT

  • Import or synchronization flows
  • Background processing workflows
  • Validation and enrichment sequences
  • Operational scripts and integration calls

NOT THE GOAL

  • Not a visual workflow designer
  • Not distributed orchestration
  • Not enterprise BPM
  • Use it when the work is meaningfully step-based
STEP-BASED

ordered work

RESULT

carried state

TRACKED

runtime visibility

KEY IDEA

Pipelines make workflow sequencing visible in one place instead of scattering order, retry, exit, and tracking logic across services and handlers.

Part 1

Authoring

Start with a pipeline context, pick an authoring style, and define the ordered steps explicitly.

Packaged Pipeline Definition

Reusable packaged pipeline

public partial class OrderImportContext : PipelineContextBase
{
    [ValidateNotEmpty("Source file name is required.")]
    public string SourceFileName { get; set; }

    public int ImportedOrderCount { get; set; }
}

public class OrderImportPipeline : PipelineDefinition<OrderImportContext>
{
    protected override void Configure(IPipelineDefinitionBuilder<OrderImportContext> builder)
    {
        builder
            .AddStep<ValidateOrderImportStep>()
            .AddStep<LoadOrdersStep>()
            .AddStep<PersistOrdersStep>()
            .AddHook<PipelineAuditHook>()
            .AddBehavior<PipelineTracingBehavior>();
    }
}

DEFAULT CHOICE

  • Use packaged definitions when a flow belongs clearly to one feature
  • Keep shared workflow state on the context type
  • Use hooks and behaviors during definition, not buried inside steps

CONTEXT FIRST

The context is the shared execution state. If transient data needs to flow across steps, it belongs on the context instead of in hidden locals or service fields.

Basic Execution Flow

Resolve and execute a pipeline

var pipeline = pipelineFactory.Create<OrderImportPipeline, OrderImportContext>();

var context = new OrderImportContext
{
    SourceFileName = "orders.csv"
};

var result = await pipeline.ExecuteAsync(
    context,
    options => options
        .ContinueOnFailure()
        .MaxRetryAttemptsPerStep(3));

FLOW

  • Resolve a named or packaged pipeline from IPipelineFactory
  • Pass a concrete context object into execution
  • Configure per-run execution options at the call site

RESULT MODEL

Awaited execution returns the carried Result. The pipeline abstraction is about workflow control, not about inventing a second success/failure model.

Context Validation And Authoring Options

CONTEXT VALIDATION

  • Validation attributes and static [Validate] methods can live on the context type
  • Validation runs before hooks, behaviors, and steps
  • Add the code-generation analyzer package where pipeline contexts live

AUTHORING OPTIONS

  • PipelineDefinition<TContext> for reusable pipelines
  • PipelineDefinitionBuilder<TContext> for programmatic definitions
  • AddPipelines().WithPipeline(...) for small local flows

Packaged definitions are the best default. Builder and inline registration are there when the pipeline is smaller, dynamic, or tightly local to composition setup.

Builder And Inline Registration

Direct builder and DI-time registration

var definition = new PipelineDefinitionBuilder<OrderImportContext>("order-import")
    .AddStep<ValidateOrderImportStep>()
    .AddStep<LoadOrdersStep>()
    .AddStep(context => context.ImportedOrderCount++)
    .AddAsyncStep(async execution =>
    {
        await Task.Yield();
        return execution.Continue(execution.Result.WithMessage("Inline step finished."));
    })
    .Build();

services.AddPipelines()
    .WithPipeline<OrderImportContext>("order-import", builder => builder
        .AddStep<ValidateOrderImportStep>()
        .AddBehavior<PipelineTracingBehavior>());

WHEN TO USE

  • Builder for programmatic definition composition
  • Inline registration for small feature-local workflows
  • Inline steps for small orchestration fragments, not large business flows

Part 2

Execution

Decide whether to await the result, run in the background, or customize retry and continuation behavior per execution.

Awaited And Background Execution

Await the result or track background execution

var result = await pipeline.ExecuteAsync(
    context,
    options => options
        .ContinueOnFailure()
        .MaxRetryAttemptsPerStep(5));

var handle = await pipeline.ExecuteAndForgetAsync(
    context,
    options => options.WhenCompleted(completion =>
    {
        Console.WriteLine($"Pipeline {completion.ExecutionId} finished with {completion.Status}");
        return ValueTask.CompletedTask;
    }));

var snapshot = await tracker.GetAsync(handle.ExecutionId);

AWAITED

  • Best when the caller needs the final result now
  • Natural fit for in-request orchestration

BACKGROUND

  • Best for long-running flows that should return quickly
  • Execution handle plus tracker snapshot becomes the query surface

Flow Control Through PipelineControl

CONTINUE

Move to the next step

default success path and final completion path

RETRY

Repeat the same step attempt

bounded by MaxRetryAttemptsPerStep

BREAK

Finish early in a controlled way

use when there is intentionally no more work to do

TERMINATE

Stop immediately

use when the flow should be blocked or intentionally ended

FAILURES AND EXCEPTIONS

Exceptions are caught and appended as errors to the carried Result. Normal control flow should still prefer explicit PipelineControl and result-based failure state.

Hooks Versus Behaviors

HOOKS

  • Observe lifecycle events
  • Pipeline start, step start, step completion, pipeline completion/failure
  • Good for auditing and notifications

BEHAVIORS

  • Wrap execution around the whole pipeline and each step attempt
  • Good for tracing, timing, scopes, and other cross-cutting concerns
  • Behaviors affect execution; hooks observe it

If the concern should wrap execution, use a behavior. If it should listen to lifecycle moments, use a hook.

Tracking, Logging, And Tracing

TRACKING

  • IPipelineExecutionTracker stores background execution snapshots
  • Track status, current step, timestamps, and latest carried result
  • Use it as the canonical query surface for background runs

OBSERVABILITY

  • Internal logging uses pipeline and step names rather than CLR type names
  • PipelineTracingBehavior creates one outer activity plus nested step activities
  • Progress can flow through PipelineExecutionOptions

Part 3

Advanced Usage

Mix class-based and inline steps, keep naming predictable, and use code generation when packaged pipeline boilerplate starts to dominate.

Step Authoring Styles

Class-based and inline steps

public class PersistOrdersStep : AsyncPipelineStep<OrderImportContext>
{
    protected override async ValueTask<PipelineControl> ExecuteAsync(
        OrderImportContext context,
        Result result,
        PipelineExecutionOptions options,
        CancellationToken cancellationToken)
    {
        await repository.SaveAsync(context.ImportedOrderCount, cancellationToken);
        return PipelineControl.Continue(result.WithMessage("Orders persisted."));
    }
}

builder.AddAsyncStep(async execution =>
{
    if (execution.Result.IsFailure)
    {
        return execution.Break();
    }

    await repository.SaveAsync(execution.CancellationToken);
    return execution.Continue(execution.Result.WithMessage("Saved from inline step."));
});

CLASS-BASED

  • Better for reusable or non-trivial workflow logic
  • Clear separation and testability

INLINE

  • Better for small local orchestration fragments
  • Use the execution object when you need services, cancellation, or control helpers

Pipeline Code Generation

Generated packaged pipeline authoring

[Pipeline(typeof(OrderImportContext))]
[PipelineHook(typeof(PipelineAuditHook))]
[PipelineBehavior(typeof(PipelineTracingBehavior))]
public partial class OrderImportPipeline
{
    [PipelineStep(10)]
    public Result Validate(OrderImportContext context, Result result)
    {
        return result.WithMessage("validated");
    }

    [PipelineStep(20)]
    public async Task<Result> LoadAsync(
        OrderImportContext context,
        Result result,
        IOrderImportRepository repository,
        CancellationToken cancellationToken)
    {
        var orders = await repository.LoadAsync(context.SourceFileName, cancellationToken);
        context.ImportedOrderCount = orders.Count;
        return result.WithMessage($"loaded {orders.Count} orders");
    }
}

WHAT IT ADDS

  • [Pipeline] removes packaged definition boilerplate and the explicit base type
  • [PipelineStep(order)] for generated step wiring
  • Class-level hooks and behaviors stay declarative

WHEN TO USE

Use code generation when the packaged pipeline shape is right but hand-authoring the definition plumbing is adding more ceremony than value.
