Requester And Notifier

Guide to dispatching requests and notifications through shared pipelines, reusable behaviors, and source-generated message authoring.

Based on features-requester-notifier.md, with related references to features-results.md.

Requester And Notifier At A Glance

01 / REQUESTER

One message. One handler.

  • Commands and queries
  • Dispatch through IRequester
  • Return Result<T>

02 / NOTIFIER

One message. Many handlers.

  • Publish/subscribe notifications
  • Dispatch through INotifier
  • Return aggregated Result

03 / SHARED PIPELINE

Cross-cutting concerns stay outside handlers

  • Validation
  • Retry, timeout, chaos, authorization
  • Progress reporting and metadata

Core Architecture

1

Caller sends or publishes

IRequester or INotifier

->
2

Pipeline behaviors run

validation, retry, timeout, auth, custom logic

->
3

Handlers execute

one request handler or multiple notification handlers

->
4

Explicit result returns

Result<T> or Result

KEY IDEA

Both systems standardize message dispatching, error propagation, metadata, and extensibility without forcing technical concerns into handler bodies.

Part 1

Requester

Single-handler request/response flow for commands and queries.

Requester Mental Model

COMMANDS

  • Modify state
  • Usually return Result<Unit> or a small response model
  • One request maps to one primary business operation

QUERIES

  • Read data without changing state
  • Return Result<T>
  • Still use the same requester pipeline
1

handler

TYPE-SAFE

request + response

TRACKED

request id + timestamp

Basic Request Flow

Requester dispatch and handler

public class DoSomethingCommand : RequestBase<Unit>
{
    public string Message { get; set; }
}

public class DoSomethingCommandHandler
    : RequestHandlerBase<DoSomethingCommand, Unit>
{
    protected override async Task<Result<Unit>> HandleAsync(
        DoSomethingCommand request,
        SendOptions options,
        CancellationToken cancellationToken)
    {
        await Task.Delay(100, cancellationToken);
        return Result<Unit>.Success(Unit.Value);
    }
}

var result = await requester.SendAsync(new DoSomethingCommand
{
    Message = "Performing task..."
});

FLOW

  • Caller creates request message
  • IRequester.SendAsync(...) dispatches it
  • One handler returns a typed result

VALIDATORS

Nested FluentValidation validators are discovered automatically and can run through the normal validation behavior.

Policies, Metadata, And SendOptions

METADATA

Built into RequestBase<T>

RequestId and RequestTimestamp

SEND OPTIONS

Control dispatch-time behavior

progress reporting, exception handling, cancellation

HANDLER POLICIES

Attach on the handler surface

retry, timeout, circuit breaker, chaos, transaction

BEHAVIOR OPTIONS

Configure globally

fluent API for defaults and priorities

Requester gives you both local control at the message/handler level and central control through builder configuration.

Source-Generated Commands And Queries

[Command] and [Query]

[Command]
public partial class DoSomethingCommand
{
    [ValidateNotEmpty("Message cannot be empty.")]
    public string Message { get; set; }

    [Handle]
    private async Task<Result<Unit>> HandleAsync(CancellationToken cancellationToken)
    {
        await Task.Delay(100, cancellationToken);
        return Success();
    }
}

[Query]
public partial class GetUserQuery
{
    public Guid UserId { get; set; }

    [Handle]
    private async Task<Result<User>> HandleAsync(IGenericReadOnlyRepository<User> userRepository, CancellationToken cancellationToken)
    {
        var user = await userRepository.FindOneAsync(UserId, cancellationToken: cancellationToken);
        return user != null ? Success(user) : Failure($"User with ID {UserId} not found.");
    }
}

AUTHORING MODEL

  • Single partial type instead of separate message + handler classes
  • [Handle] methods infer result type
  • Simple validation via attributes, richer rules via [Validate]

STILL THE SAME PIPELINE

Source-generated requests still run through the standard Requester validation and policy behaviors.

Part 2

Notifier

Publish/subscribe notification flow with aggregated outcomes.

Notifier Mental Model

PUB/SUB

  • One notification can trigger many handlers
  • Publisher does not need to know subscribers
  • Cross-cutting behaviors are shared with Requester

RESULT MODEL

  • Returns non-generic Result
  • Aggregates outcomes from all handlers
  • Supports sequential, concurrent, and fire-and-forget execution
N

handlers

EXECUTION

sequential / concurrent / fire-and-forget

TRACKED

notification id + timestamp

Basic Notification Flow

Publish one notification to multiple handlers

public class UserRegisteredNotification : NotificationBase
{
    public Guid UserId { get; set; }
    public string Email { get; set; }
}

public class SendWelcomeEmailHandler
    : NotificationHandlerBase<UserRegisteredNotification>
{
    protected override async Task<Result> HandleAsync(
        UserRegisteredNotification notification,
        PublishOptions options,
        CancellationToken cancellationToken)
    {
        await Task.Delay(100, cancellationToken);
        return Result.Success();
    }
}

public class AuditRegistrationHandler
    : NotificationHandlerBase<UserRegisteredNotification>
{
    protected override Task<Result> HandleAsync(
        UserRegisteredNotification notification,
        PublishOptions options,
        CancellationToken cancellationToken)
    {
        return Task.FromResult(Result.Success());
    }
}

var result = await notifier.PublishAsync(new UserRegisteredNotification
{
    UserId = Guid.NewGuid(),
    Email = "user@example.com"
});

FLOW

  • Caller creates one notification message
  • INotifier.PublishAsync(...) dispatches it to all matching handlers
  • Each handler returns a Result instead of a response payload

WHY IT FITS

Use this shape for side effects like audit, email, cache invalidation, metrics, and other reactions that should stay decoupled from the publisher.

Notification Flow And Result Aggregation

1

Publisher calls PublishAsync(...)

dispatch notification

->
2

Behaviors run

validation and shared cross-cutting logic

->
3

Multiple handlers execute

one after another, in parallel, or fire-and-forget

->
4

Aggregate result returns

success or accumulated failures

USE IT FOR

Reactions like email, logging, cache updates, metrics, and other side effects where multiple subscribers should be able to respond independently.

Execution Modes

SEQUENTIAL

Ordered and fail-fast

Default mode. Runs handlers one after another and stops on the first failure.

CONCURRENT

Parallel fan-out

Runs handlers independently and aggregates the outcomes.

FIRE-AND-FORGET

Returns immediately

Invokes handlers without waiting for completion when the publish call should not block.

Execution mode is part of the design choice, not just a performance toggle. Pick it based on ordering, isolation, and whether the publisher depends on completion.

Source-Generated Events

[Event] with generated and manual handlers

[Event]
[HandlerRetry(2, 300)]
public partial class UserRegisteredEvent
{
    [ValidateNotEmpty("Email is required.")]
    [ValidateEmail("Email must be valid.")]
    public string Email { get; init; }

    [Handle]
    private Result Audit()
    {
        Console.WriteLine($"Audit user registration for {Email}");
        return Success();
    }

    [Handle]
    private async Task<Result> SendWelcomeEmailAsync(
        IEmailService emailService,
        CancellationToken cancellationToken)
    {
        await emailService.SendAsync(Email, cancellationToken);
        return Success();
    }
}

EVENT AUTHORING

  • One partial type can declare one or more [Handle] methods
  • Each handle becomes a generated notification handler
  • Generated handlers do not replace manual subscribers

VALIDATION

Property validators and optional [Validate] methods still run through the standard Notifier validation pipeline.

Part 3

Pipeline Behaviors

Shared cross-cutting logic for both Requester and Notifier.

Shared Behavior Model

VALIDATION

Stop invalid messages early

nested validators and generated validators plug in here

RETRY / TIMEOUT

Protect critical operations

can be configured globally or per handler

AUTH / CHAOS / CACHE

Reusable policy layer

shared behavior surface for technical concerns

CUSTOM

Extend with your own logic

write once and apply to both systems

Behaviors are registered through the fluent builder and run in registration order, so ordering is part of the design.

Custom Behaviors And Progress Reporting

Custom behavior sketch

public class LoggingPipelineBehavior<TRequest, TResponse>
    : PipelineBehaviorBase<TRequest, TResponse>
    where TRequest : class
{
    protected override async Task<TResponse> HandleAsync(
        TRequest request,
        object options,
        Type responseType,
        Func<Task<TResponse>> next,
        CancellationToken cancellationToken)
    {
        Console.WriteLine($"Handling {typeof(TRequest).Name}");
        return await next();
    }
}

CUSTOM BEHAVIORS

  • One implementation can serve both Requester and Notifier
  • Ideal for logging, correlation, tracing, and other cross-cutting concerns

PROGRESS

  • IProgress<ProgressReport> can flow through SendOptions and PublishOptions
  • Useful for long-running operations without polluting handler contracts

Built-In Policies And Behavior Options

PER-HANDLER ATTRIBUTES

  • [HandlerRetry]
  • [HandlerTimeout]
  • [HandlerCircuitBreaker]
  • [HandlerChaos]
  • [HandlerDatabaseTransaction]

BUILDER OPTIONS

  • Configure defaults globally
  • Chain behaviors fluently
  • Control priorities and override behavior centrally
  • Separate Requester and Notifier options where needed

The feature gives you both tactical per-handler policies and strategic global defaults. Use both intentionally.

Part 4

Advanced Usage

Generic handlers, comparisons, and adoption guidance.

Generic Handlers, CQS, And Adoption Notes

GENERIC HANDLERS

  • AddGenericHandlers() discovers open generic handlers
  • Registers closed handlers when constraints are satisfied
  • Works for both Requester and Notifier

ADOPTION NOTES

  • Use Requester when one caller needs one answer
  • Use Notifier when one message should fan out to many subscribers
  • Keep handlers small and let behaviors carry the shared technical concerns

COMPARISON

Compared with MediatR-style setups, this feature bakes in results, metadata, notification execution modes, and behavior/policy conventions instead of leaving those concerns to separate custom infrastructure.
